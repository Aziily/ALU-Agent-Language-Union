# v0.7 Pipeline C — Pilot Decision Report

**Date**: 2026-05-18
**Author**: Claude (autonomous loop)
**Model**: gpt-5.4 via local :9000 proxy
**Token budget used**: ~780k across 5 pilot cells

## TL;DR

- **Pipeline C (greenfield AL authoring) is viable.** It hits baseline-parity
  on cachetools (95.8% per-test pass best-iter, both with and without v0.7
  rich features) and survives projects where Pipeline B catastrophically
  fails (wcwidth: B times out on the LLM call due to massive Unicode-table
  re-emission; C reaches 71.8%).
- **The LLM CAN adopt v0.7 advanced features** (intent, typed I/O,
  top-level relative imports) when shown a few-shot example — feature
  usage jumped from 0 to ~10-13 instances per file. But the test pass%
  is unchanged. v0.7's editorial vocabulary is **adoption-friendly but
  benchmark-neutral**.
- **One real parser bug found and fixed**: v0.7 didn't accept Python-style
  relative imports (`from . import X`). Found because the LLM started
  emitting them once the few-shot example demonstrated top-level imports.
  Fixed in commit [d711c63].
- **One real fairness bug found and fixed**: Pipeline C had max_tokens=16384
  vs A/B's 12288. Fixed in commit [3f74321]. (Empirically harmless — no
  cell ever hit the cap — but the asymmetry is gone.)

## Per-cell results

| Project | A baseline | B al-skel | C al-greenfield | Token cost |
|---|---|---|---|---|
| cachetools | best=95.8%, final=95.8% | best=96.3%, final=96.3% | best=95.8%, final=82.8% | A=35k, B=75k, **C=52k** |
| cachetools (rich C) | — | — | best=96.3%, final=96.3% | C=56k |
| wcwidth | best=69.2%, final=69.2% | **❌ ReadTimeout** | best=71.8%, final=7.7% | A=192k, B=0, C=208k |

**Key observations**:

1. **Mid-run regression**: both A and C show iter N+1 < iter N's best in some
   cases (cachetools C: 95.8% → 82.8%, wcwidth C: 71.8% → 7.7%). gpt-5.4
   at temp=0 is not deterministic enough for feedback loop to monotonically
   improve. **Recommendation**: report best-iter as headline, not final-iter.
   The harness already tracks `best_iter_idx` / `best_iter_pass_pct`.

2. **Pipeline B's skeleton roundtrip tax**: B's prompt is 1.76× A's, B's
   completion is 5.75× A's (for cachetools). B has to re-emit the full
   .al including verbatim `body:` Python in every preamble. For wcwidth
   the Unicode tables embedded in skeleton constants blow this up enough
   to time out the LLM call. **Pipeline C avoids this** because it doesn't
   need to re-emit unchanged source — the model only writes the new bodies.

3. **A vs C is the fairest comparison**. Both receive the same input
   (stripped Python + spec). B receives a privileged input (hand-written
   skeleton with all preambles + imports + constants pre-done — the LLM
   only fills bodies). The original "往返税" (A vs B Δ) measures
   *scaffold + AL output overhead combined*, not AL language tax alone.

## What the LLM actually wrote in greenfield mode

**Without few-shot prompt** (pilot 1, cachetools):
- Used: `preamble` with `source`/`constants`/`imports` block + `code` with `body`
- Used: cross-file via Python imports inside `imports:` block (NOT top-level)
- Did NOT use: `intent:`, `input:`, `output:` (typed annotation), top-level
  AL `import`/`from`, `flow`, `agent`, `set`, `return`
- Effective treatment of AL: "Python container with file boundaries"

**With v0.7 few-shot prompt** (pilot 4, cachetools, parser fixed):
- intent: 12 / 13 nodes
- input: 10 / 12 code nodes
- output: 10 / 12 code nodes
- top-level `from X import Y`: 3
- Body Python: same quality (96.3% test pass)
- Token cost: +9% (56k vs 51k) — acceptable

The v0.7 features are **acquired** via prompting, not **emergent** from
the language design alone. This is a meaningful design insight: AL's
syntax tells the model what's POSSIBLE, but few-shot examples are needed
to tell the model what's RECOMMENDED.

## Fairness audit (final)

| Aspect | A | B | C | Fair? |
|---|---|---|---|---|
| LLM client / temperature=0 / MAX_ITER=3 / max_tokens=12288 | same | same | same | ✅ |
| Test runner / workdir revert | same | same | same | ✅ |
| Spec text / iter feedback format | same | same | same | ✅ |
| Input | stripped .py | **hand .al skeleton** | stripped .py | ⚠ B privileged |
| Authoring guide | ❌ none | ✅ given | ✅ given | by design |
| Output target | filled .py | filled .al (byte-id) | new multi-file .al | by design |

**Conclusions**:
- A vs C is the cleanest experiment for "does AL help LLM generate code".
- B is a separate research question: "does scaffold help".
- Future cross-pipeline comparisons should report A/B/C separately, not
  as a single "往返税" number, because the asymmetry is now load-bearing.

## Out of scope / Known limitations

- Pipeline C currently uses inject_filled_al per file (NOT the v0.7 codegen
  multi-file emit). That's fine for commit0 (each .al's body Python lands
  in the matching .py), but it means the v0.7 `import` / `from` declarations
  are essentially documentation, not load-bearing at runtime. To make them
  load-bearing, would need codegen → write out .py files via `emit_project`.
- mid-run regression mitigation (rollback to best iter) is a methodology
  change; current code uses last iter's workdir state.
- Pipeline C's prompt doesn't yet enforce `intent:`/`input:`/`output:` as
  HARD requirements; they're suggested. To test "AL forces better
  thinking" we'd need to fail-on-missing those fields.
